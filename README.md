# Домашнее задание к занятию "`Репликация и масштабирование. Часть 2`" - `Селезнев Александр`

### Задание 1

`Опишите основные преимущества использования масштабирования методами:

- активный master-сервер и пассивный репликационный slave-сервер;
- master-сервер и несколько slave-серверов;`

`1. Активный master-сервер и пассивный репликационный slave-сервер

В данной архитектуре все операции записи выполняются на master-сервере, а slave-сервер постоянно получает изменения посредством репликации и находится в режиме ожидания.

Преимущества:

- высокая отказоустойчивость — при выходе из строя master можно переключиться на slave;
- наличие резервной копии данных в режиме, близком к реальному времени;
- возможность сократить время простоя при аварии;
- возможность выполнять резервное копирование с реплики, не нагружая основной сервер;
- простота настройки и сопровождения по сравнению с более сложными кластерами.

Недостатком является то, что в обычном режиме slave практически не используется для обработки пользовательских запросов.

2. Master-сервер и несколько slave-серверов

В этой архитектуре запись выполняется только на master-сервер, а запросы на чтение распределяются между несколькими slave-серверами.

Преимущества:

- увеличение производительности системы за счет распределения запросов на чтение;
- снижение нагрузки на основной сервер;
- возможность масштабирования при увеличении количества пользователей;
- возможность обслуживать аналитические запросы и отчеты на отдельных репликах;
- повышение отказоустойчивости - при отказе одной реплики остальные продолжают работать.

Такая схема хорошо подходит для систем, где операций чтения значительно больше, чем операций записи (например, интернет-магазины, библиотеки, новостные сайты).`

### Задание 2

`Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трех таблиц:

- пользователи,
- книги,
- магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.`

`План горизонтального и вертикального шардинга базы данных

Пусть база данных состоит из следующих таблиц:

- Пользователи (users)
- Книги (books)
- Магазины (stores)

Вертикальный шардинг

При вертикальном шардинге разные таблицы располагаются в разных базах данных.

База данных №1
users

Содержит информацию о пользователях:

- id
- ФИО
- email
- пароль
- телефон

База данных №2
books

Содержит информацию о книгах:

- id
- название
- автор
- ISBN
- цена

База данных №3
stores

Содержит информацию о магазинах:

- id
- название
- адрес
- телефон
- график работы

Преимущества
- независимое масштабирование каждой базы;
- уменьшение нагрузки между серверами;
- возможность использовать разные параметры хранения;
- упрощение обслуживания отдельных частей системы.

Горизонтальный шардинг
При горизонтальном шардинге одна и та же таблица делится по строкам между несколькими серверами.

Например, таблицу users можно разделить следующим образом:

Shard 1

```id 1–1 000 000```

Shard 2

```id 1 000 001–2 000 000```

Shard 3

```id более 2 000 000```

Либо распределять пользователей по региону:

- Европа
- Азия
- Америка

Точно так же можно разделить таблицы books и stores.

Принципы построения системы
- приложение обращается к балансировщику запросов;
- балансировщик определяет нужный сервер;
- запись выполняется только на основной сервер соответствующего шарда;
- чтение может выполняться с реплик;
- каждый шард имеет собственную резервную реплику.

Такой подход позволяет увеличивать производительность практически линейно при росте нагрузки.`

`Расположение данных:

- Каждый шард – это независимый сервер (или кластер) базы данных.

- Внутри шарда может быть настроена репликация Master-Slave для отказоустойчивости и чтения.

- Сервис пользователей обращается к своим шардам через логику маршрутизации (по user_id).

- Сервисы книг и магазинов работают аналогично.

Режимы работы серверов
1. Основной режим – Master-Slave внутри каждого шарда
- Master принимает все операции записи и изменения, реплицирует их на Slave.
- Slave(ы) обслуживают запросы чтения, используются для снятия бэкапов.
- При отказе Master’а один из Slave повышается до Master (автоматический failover через Patroni / Orchestrator).

Такой режим гарантирует надёжность хранения и возможность масштабирования чтения даже на уровне отдельного шарда.

2. Шарды работают независимо
Каждый шард является изолированным хранилищем своего подмножества данных. При выходе из строя одного шарда остаются недоступными только пользователи/книги/магазины этого шарда, а не вся система (частичная деградация).

3. Сервисы взаимодействуют через асинхронные сообщения или API
Для поддержания консистентности между доменами (например, при оформлении заказа) используется паттерн Saga или событийная модель, что позволяет избегать распределённых транзакций и блокировок.

Этот план обеспечивает гибкое масштабирование как по объёму данных, так и по нагрузке, сохраняя высокую доступность и отказоустойчивость.`
